Full discussion of all 23 design patterns based on the cheatsheet

Creational Patterns — deal with object creation
PatternIntent + DiscussionKeywords / Example
Singleton
Creational		 Ensures a class has only one instance and provides a global access point to it. The constructor is made private; a static method returns the single instance, creating it on first call.
e.g. Database connection, app config manager, logger — anywhere one shared instance must coordinate the whole system.
 only one instance · shared resource · global access point
Prototype
Creational		 Lets you copy/clone existing objects without making your code depend on their concrete class. Instead of new ConcreteClass(), you call clone() on an existing object.
e.g. Duplicating a complex game character with all its attributes — cheaper than rebuilding from scratch.
 copy / clone · costly creation · pre-built templates
Builder
Creational		 Lets you construct complex objects step by step, producing different representations using the same construction code. Separates construction logic from the object itself. Uses a fluent .withX().build() chain.
e.g. Building a SQL query, a meal order (burger + drink + side), or an HTTP request with many optional parameters.
 step by step · many optional params · .withX().build()
Factory Method
Creational		 Provides an interface for creating objects, letting subclasses decide which concrete type to instantiate. The parent class defines the factory method; subclasses override it. Client code never calls new directly.
e.g. A logistics app with createTransport() — subclasses return Truck or Ship. Adding a new type = new subclass only.
 indirect creation · hides creation complexity · supports new types
Abstract Factory
Creational		 Lets you produce families of related objects together without specifying their concrete classes. Think of it as a "factory of factories." Swapping the whole factory swaps the entire product family at once.
e.g. UI toolkit: a WindowsFactory creates Windows-style buttons + checkboxes; a MacFactory creates Mac-style ones. Swap the factory = swap the whole "kit".
 family of related objects · compatible product set · swap the whole "kit"
Structural Patterns — deal with object composition
PatternIntent + DiscussionKeywords / Example
Adapter
Structural		 Allows objects with incompatible interfaces to work together via a wrapper that converts one interface to another. The Adapter implements the target interface and holds a reference to the Adaptee.
e.g. A third-party payment API expects pay(cardNumber) but your code calls processPayment(userId) — Adapter bridges the gap.
 incompatible interface · third-party library · reconcile two interfaces
Bridge
Structural		 Decouples an abstraction from its implementation so the two can vary independently. Instead of inheriting implementations, the abstraction holds a reference to an implementor interface.
e.g. A Shape (abstraction) + Color (implementation) — you can have RedCircle, BlueCircle without a class explosion.
 two independent dimensions · separate abstraction from implementation · avoid class multiplication
Composite
Structural		 Lets you compose objects into tree structures and treat individual objects and groups uniformly. Both leaf and composite nodes implement the same interface.
e.g. A file system: File and Folder both have getSize(). A folder's size = sum of its children's sizes.
 tree structure · part-whole · treat individual and group uniformly
Decorator
Structural		 Lets you add behaviors/responsibilities to an object dynamically by wrapping it in decorator objects, without modifying its class. Decorators implement the same interface as the object they wrap.
e.g. Coffee orders: SimpleCoffee wrapped with MilkDecorator, then SugarDecorator — each adds cost + description.
 add behavior without subclassing · dynamically · stacks/layers · wrap the object
Facade
Structural		 Provides a single simplified interface to a complex subsystem of classes. The facade doesn't add new functionality — it just makes the subsystem easier to use.
e.g. A HomeTheaterFacade.watchMovie() that internally turns on TV, dims lights, starts the sound system.
 simplify a complex subsystem · one simple interface · hide complexity
Flyweight
Structural		 Reduces memory usage by sharing common (intrinsic) state across a large number of objects. Intrinsic state = shared, stored in flyweight. Extrinsic state = unique per object, passed in at runtime.
e.g. A forest with 1,000,000 trees — the tree type/texture (intrinsic) is shared; only x/y position (extrinsic) differs per tree.
 thousands of objects · save memory · intrinsic (shared) vs extrinsic (varying)
Proxy
Structural		 Provides a substitute/placeholder for another object to control access to it. The proxy implements the same interface as the real object and adds logic before/after delegating.
Types: Protection Proxy (API key check), Virtual Proxy (lazy-load), Caching Proxy (cache results).
 control access · same interface · validate before delegating · lazy-load / caching
Behavioral Patterns — deal with object communication
PatternIntent + DiscussionKeywords / Example
Chain of Responsibility
Behavioral		 Passes a request along a chain of handlers; each handler decides either to process it or pass it to the next. Decouples sender from receiver.
e.g. Support tickets: Level1 → Level2 → Level3. Each level handles if it can, otherwise forwards.
 pass request along a chain · multiple handlers · each handles or forwards
Command
Behavioral		 Encapsulates a request as a standalone object, enabling you to pass, queue, log it, and support undo. Each command has an execute() and optionally undo().
e.g. Text editor undo/redo, task schedulers, transaction logs.
 undo support · encapsulate request as object · queue/schedule operations
Interpreter
Behavioral		 Defines a representation for a language's grammar along with an interpreter that uses it to interpret sentences in that language.
e.g. SQL parsers, regex engines, math expression evaluators.
 interpret/translate a language · grammar rules · recurring expressions
Iterator
Behavioral		 Lets you traverse elements of a collection sequentially without exposing its internal structure. Each collection provides its own iterator.
e.g. Java's for-each loop, iterating over arrays, trees, graphs all with the same next() interface.
 traverse a collection · without exposing internal structure · traverse / next()
Mediator
Behavioral		 Reduces chaotic coupling between many objects by making them communicate through a central mediator object instead of directly. Objects don't know about each other.
e.g. Air traffic control: planes don't talk to each other, they all communicate through the control tower.
 central communication hub · reduce coupling · instead of direct talk
Memento
Behavioral		 Lets you save and restore an object's internal state without exposing its details — to support undo or snapshots. Three roles: Originator (has state), Memento (stores snapshot), Caretaker (manages history).
e.g. Undo in Photoshop, game save/load checkpoints.
 save/restore state · snapshot · checkpoints · undo
Observer
Behavioral		 Defines a one-to-many relationship so that when one object (Subject) changes state, all its subscribers (Observers) are notified automatically. Subject has attach(), detach(), notifyObservers().
e.g. MVC — Model notifies Views. News feed notifying all subscribers. Player score notifying fans.
 notify multiple objects · subscribe/unsubscribe · one-to-many · broadcast updates
State
Behavioral		 Lets an object alter its behavior when its internal state changes, as if it changed its class. Each state is a separate class implementing a common interface.
e.g. A vending machine: IdleState, HasMoneyState, DispensingState — each responds differently to the same events.
 behavior changes with state · state machine · transitions between states
Strategy
Behavioral		 Defines a family of algorithms, encapsulates each, and makes them interchangeable at runtime. The client holds a reference to a strategy interface and can swap it without changing its own code.
e.g. Payment methods (Credit/PayPal/Cash), sorting algorithms, file parsers (JSON/XML/CSV).
 select algorithm at runtime · different interchangeable algorithms · dynamically
Template Method
Behavioral		 Defines the skeleton of an algorithm in a parent class, deferring some steps to subclasses to redefine without changing the overall structure. The parent calls abstract methods that children implement.
e.g. Report generation: fetch → calculate → format → save. The parent fixes the order; only calculate() is overridden per subclass.
 fixed-structure process · override steps only · same sequence · template fixes the order
Visitor
Behavioral		 Lets you add new operations to a set of objects without modifying their classes, by separating the operation from the structure. Each element accepts a visitor; the visitor contains the new operation logic.
e.g. A document exporter: nodes (Paragraph, Image, Table) accept an ExportVisitor without changing their own classes.
 add a new operation without changing classes · separate operation from structure · temporary extra behavior
